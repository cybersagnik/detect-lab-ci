# T1003.001 — LSASS Memory Access (Credential Dumping)
 
## Hypothesis
Adversaries opening a handle to `lsass.exe` with memory-read access
to extract credential material (NTLM hashes, Kerberos tickets, cleartext
passwords) using tools such as Mimikatz or ProcDump.
 
## Threat actor context
Used by: virtually every threat actor performing lateral movement — Lazarus
Group, APT28, APT29, FIN6, and all commodity ransomware families. LSASS
credential dumping is present in the vast majority of enterprise ransomware
incidents and is the primary technique for obtaining credentials after
initial access.
 
## Telemetry required
- **Sysmon EventID 10** (ProcessAccess)
- `TargetImage`: `lsass.exe`
- `GrantedAccess`: specific masks associated with memory reading:
  - `0x1010` — PROCESS_VM_READ + PROCESS_QUERY_INFORMATION
  - `0x1410` — common Mimikatz mask
  - `0x1438` — common dumping mask
  - `0x143a` — Mimikatz variant
  - `0x1fffff` — PROCESS_ALL_ACCESS (almost never legitimate)
  - `0x40` — PROCESS_DUP_HANDLE (used by some dumpers)
## Why standard detection misses this
Monitoring all LSASS access events is very noisy — Defender, backup
software, and multiple system processes legitimately open LSASS handles.
Filtering by SourceImage alone is insufficient because:
1. Process names can be spoofed
2. Legitimate processes can be injected and used as dumping vehicles
3. A compromised MsMpEng.exe or svchost.exe opening LSASS should still alert
## Detection logic
```yaml
detection:
  selection:
    TargetImage|endswith: '\lsass.exe'
    GrantedAccess|contains:
      - '0x1010'
      - '0x1410'
      - '0x1438'
      - '0x143a'
      - '0x1fffff'
      - '0x40'
  filter_wininit:
    SourceImage|endswith: '\wininit.exe'
  condition: selection and not filter_wininit
```
 
## Lab validation
- ART Test: `Invoke-AtomicTest T1003.001 -TestNumbers 1`
- Alert fired: Yes — Sysmon EID 10
- TargetImage confirmed: `C:\Windows\System32\lsass.exe`
- GrantedAccess mask `0x1010` observed in alert
## Tuning history
 
| Version | FP Count | Change |
|---|---|---|
| v1 | 30 | Filtered by SourceImage: svchost, MsMpEng, taskmgr, wininit — all except wininit are common injection targets |
| v2 | 1 | Switched to GrantedAccess mask filter. Only wininit.exe filtered by process name |
 
## Filter rationale
 
**GrantedAccess mask approach (chosen):**
Legitimate system processes opening LSASS for their own purposes
(session management, authentication) use different access masks that
do not include `PROCESS_VM_READ`. Filtering on the specific masks
required for memory dumping is more resilient than process name filtering.
 
**Only `wininit.exe` filtered by process name:**
- Initialises LSASS at boot — holds a permanent legitimate handle
- Runs from protected `C:\Windows\System32\` — not writable by user processes
- Cannot be spawned by user-level processes without elevation
- No known exploitation vector specifically for LSASS access
**`svchost.exe` and `MsMpEng.exe` intentionally NOT filtered:**
Both are primary targets for process injection. A compromised instance
requesting credential-dumping access masks is exactly what this rule
should detect — filtering them would create a significant blind spot.
 
## Accepted FP count: 1
One legitimate system process opened LSASS with a monitored access mask
on the clean baseline. At `critical` severity this is intentional — each
alert should be investigated. The GrantedAccess mask and SourceImage
together confirm or refute in under one minute.
