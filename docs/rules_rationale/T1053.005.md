# T1053.005 — Suspicious Scheduled Task Creation via schtasks.exe
 
## Hypothesis
Adversaries creating scheduled tasks via `schtasks.exe` that run
executables from user-writable locations or spawn interpreter processes
as a persistence mechanism surviving reboots and user logoffs.
 
## Threat actor context
Used by: Emotet, Qbot, FIN7, Lazarus Group, and most enterprise-targeting
ransomware families. One of the most reliable Windows persistence mechanisms
— tasks survive reboots, can run as SYSTEM, and are commonly overlooked
during initial IR triage due to volume of legitimate task creation activity.
 
## Telemetry required
- **Sysmon EventID 1** (ProcessCreate)
- Image: `schtasks.exe`
- CommandLine: `/Create` + `/TR` (task run — specifies what executes)
- **Windows Security EventID 4698** (secondary — scheduled task created)
## Why standard detection misses this
Monitoring all `schtasks /Create` events generates extreme noise — Windows
creates tasks constantly for updates, maintenance, and telemetry. Filtering
by ParentImage is dangerous (parent processes are common injection targets).
The signal must come from WHAT the task runs, not just that a task was created.
 
## Detection logic
```yaml
detection:
  selection_base:
    Image|endswith: '\schtasks.exe'
    CommandLine|contains|all:
      - '/Create'
      - '/TR'
  selection_suspicious:
    CommandLine|contains:
      - 'powershell'
      - 'cmd.exe'
      - 'wscript'
      - 'cscript'
      - 'mshta'
      - 'regsvr32'
      - '\Temp\'
      - '\AppData\Local\Temp'
      - '\AppData\Roaming'
  filter_safe:
    CommandLine|contains:
      - '/TR "C:\Windows\'
      - '/TR "C:\Program Files\'
      - '/TR "C:\Program Files (x86)\'
  condition: selection_base and (selection_suspicious or not filter_safe)
```
 
## Lab validation
- ART Test: `Invoke-AtomicTest T1053.005 -TestNumbers 1`
- Alert fired: Yes — Sysmon EID 1 and Windows Security EID 4698
- CommandLine confirmed: `schtasks /Create /SC ONCE /TR cmd.exe`
## Tuning history
 
| Version | FP Count | Change |
|---|---|---|
| v1 | 899 | `http` and `AppData` standalone strings matched every Sysmon network event (EID 3) in clean baseline via raw XML substring match |
| v2 | 9 | Removed `http` and `AppData`, added logsource EID scoping in test_runner.py to restrict evaluation to EID 1 and 4688 only |
 
**Root cause of 899 FPs:** The test runner performed raw substring match across
entire event XML. `http` appeared in every Sysmon EID 3 (NetworkConnect) event
URL field. `AppData` appeared in file path fields across hundreds of events.
Neither was in the `CommandLine` field of a `process_creation` event.
 
**Logsource scoping fix:** Added `LOGSOURCE_EVENTIDS` map in test_runner.py
that pre-filters events to only those matching the rule's logsource category
before applying detection logic. `process_creation` → EID 1, 4688 only.
 
## Accepted FP count: 9
Root cause of remaining 9 FPs: `powershell` in `selection_suspicious` matches
events where `powershell.exe` is `ParentImage` (not CommandLine content).
Two confirmed FPs are `wevtutil.exe` called from PowerShell — normal lab and
admin activity. Filtering `powershell` as a parent entirely would blind the
rule to its primary detection target (malware creating tasks that run PowerShell).
 
**Analyst triage:** Confirm Image is `schtasks.exe` and `/TR` target is a
suspicious path or interpreter. If ParentImage is a known admin tool and
the task runs from `C:\Windows\` — likely legitimate.
