# T1059.001 — Suspicious Encoded PowerShell Command
 
## Hypothesis
Adversaries encoding PowerShell commands in Base64 and executing them
via `-EncodedCommand`, `-enc`, or `-e` to obfuscate payload content
and bypass command-line string matching defences.
 
## Threat actor context
Used by: Emotet, Qbot, Lazarus Group, FIN7, APT29, and hundreds of
commodity loaders. One of the top 5 most observed execution techniques
per MITRE ATT&CK. Virtually every PowerShell-based malware family uses
encoded commands at some stage of execution.
 
## Telemetry required
- **Sysmon EventID 1** (ProcessCreate)
- Field: `CommandLine`
- Values: `-EncodedCommand`, `-enc`, `-e`
- Image: `powershell.exe` or `pwsh.exe`
- **PowerShell EventID 4104** (Script Block Logging — secondary, shows decoded payload)
## Why standard detection misses this
Basic `powershell.exe` process monitoring without CommandLine inspection
misses it entirely. The payload content is Base64 encoded so keyword
scanning of script content at rest also misses it. The `-enc`/`-e` flag
is the earliest detectable signal before the payload executes.
 
## Detection logic
```yaml
detection:
  selection:
    Image|endswith:
      - '\powershell.exe'
      - '\pwsh.exe'
    CommandLine|contains:
      - ' -EncodedCommand '
      - ' -enc '
      - '-e '
  filter_legitimate:
    CommandLine|contains:
      - 'C:\Windows\system32\WindowsPowerShell'
  condition: selection and not filter_legitimate
```
 
## Lab validation
- ART Test: `Invoke-AtomicTest T1059.001 -TestNumbers 17`
- Alert fired: Yes — Sysmon EID 1 within 2 seconds of execution
- CommandLine confirmed: `powershell.exe -e <Base64 blob>`
- EID 4104 also fired showing decoded payload content
## Tuning history
 
| Version | FP Count | Change |
|---|---|---|
| v1 | 2 | `-e` flag matched unrelated PowerShell CLI flags |
| v2 | 0 | Removed `-e` |
| v3 | 0 | Re-added `-e` — confirmed present in malicious EVTX from ART T1059.001-4 |
 
**`-e` re-added rationale:** ART test 4 uses `-e` (short form). Removing it
caused a detection miss on the actual attack sample. The 2 FPs in v1 were
from a contaminated baseline (lab session artifacts) — re-validated against
a clean idle baseline and confirmed 0 FP with `-e` present.
 
## Filter rationale
`C:\Windows\system32\WindowsPowerShell` — OS-native PowerShell automation
from its protected install path. Not writable by user processes. Low abuse
risk in this specific combined context with CommandLine restrictions.
 
## False positive rate
**Final: 0 FP** on clean Windows 10 idle baseline.
