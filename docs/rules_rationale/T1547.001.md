# T1547.001 — Registry Run Key Modification for Persistence

## Hypothesis
Adversaries writing to Windows autorun registry keys to execute malicious
code on every user login, establishing persistence after initial compromise.

## Threat actor context
Used by: virtually every commodity malware family — Emotet, AgentTesla,
RedLine, Raccoon, most RATs and infostealers. One of the oldest and most
reliable Windows persistence mechanisms. Appears in MITRE ATT&CK as one
of the top observed persistence techniques across all threat actor groups.

## Telemetry required
- **Sysmon EventID 13** (RegistryEvent — Value Set)
- `TargetObject` containing autorun key paths:
  - `\Software\Microsoft\Windows\CurrentVersion\Run`
  - `\Software\Microsoft\Windows\CurrentVersion\RunOnce`
- `Details` field — contains the value DATA being written (the executable path)
- `EventType: SetValue`

## Detection logic
```yaml
detection:
  selection:
    TargetObject|contains:
      - '\Software\Microsoft\Windows\CurrentVersion\Run'
      - '\Software\Microsoft\Windows\CurrentVersion\RunOnce'
    EventType: 'SetValue'
  filter_known_good:
    Details|contains:
      - 'OneDrive'
      - 'SecurityHealth'
      - 'Windows Defender'
      - 'Teams'
      - 'AdobeGCInvoker'
  condition: selection and not filter_known_good
```

## Lab validation
- ART Test: `Invoke-AtomicTest T1547.001 -TestNumbers 1`
- Alert fired: Yes — Sysmon EID 13
- TargetObject confirmed: `HKCU\...\CurrentVersion\Run\`

## Filter rationale — Details field approach

The `Details` field in Sysmon EID 13 contains the **value data** being
written to the registry key — in the context of Run keys, this is the
executable path or command that will run at startup.

Filtering on `Details|contains` is effective here because:

- Legitimate software writing recognisable strings (`OneDrive`, `SecurityHealth`,
  `Windows Defender`, `Teams`, `AdobeGCInvoker`) into Run key values is
  well-understood and low-risk
- These strings represent the actual payload being registered — not just
  the process name, which can be spoofed

**Why this works better than Image-based filtering on this system:**

The 139 FPs on the clean baseline came from multiple known legitimate
applications writing to Run keys. Filtering by `Image` path requires
knowing every legitimate binary path — error-prone and brittle across
different machine configurations. Filtering by `Details` content is
more portable: it describes what the Run key entry will execute, which
is what matters for determining legitimacy.

**Accepted risk:**

A sophisticated attacker could name their payload to include one of the
filtered strings (e.g. a binary called `OneDriveHelper.exe` in a malicious
path). This is a known tradeoff with value-data filtering. Mitigated by
the fact that:
1. The filtered strings are only effective bypass if the attacker knows
   exactly which strings are in the filter
2. Most commodity malware does not use these specific naming patterns
3. The filter is on partial strings — `OneDrive` would match any value
   containing that substring, so the attacker's binary name would need
   to contain it exactly

## False positive rate
**Final: 0 FP** on clean Windows 10 baseline.
Validated and confirmed working — `Details|contains` filter approach
effective on this system configuration.
